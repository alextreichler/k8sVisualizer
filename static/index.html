Guide: Redpanda ▾

Follow the official Kubernetes deployment guide step-by-step.

ⓘ Helm values → Config layers
Layer 1 — K8s objects
Values that become StatefulSet, PVC, or Service specs.
statefulset.replicas, storage.persistentVolume.size, resources.cpu.cores, resources.memory.container.max
Changes trigger a rolling restart or PVC resize.
Layer 2 — redpanda.yaml (broker config)
Helm renders these into the cm-redpanda ConfigMap, mounted at /etc/redpanda/redpanda.yaml. Read once at broker startup.
listeners.kafka.port, listeners.http.port, listeners.schemaRegistry.port, auth.sasl.enabled, tls.enabled, external.type
⚠ Requires pod restart to take effect.
Layer 3 — Admin API (live, no restart)
Applied by the post-install Job via PUT /v1/cluster_config. The configWatcher sidecar re-applies on every CR change.
config.cluster.*, config.tunable.*
Examples: log_segment_size_min, kafka_batch_max_bytes, compacted_log_segment_size
✓ Hot-reloadable. No broker restart needed.
Memory/CPU is special: affects both Layer 1 (K8s limits) and the broker CLI flags --smp and --memory passed in the StatefulSet command: array.
Tip: Click the Redpanda CR node in the graph to see the full layer breakdown with actual values.
